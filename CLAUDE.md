## АРХИТЕКТУРА СИСТЕМЫ

```
Сцена CoppeliaSim
  ├── IRB 140 (6-DOF) + вакуумный захват
  ├── Конвейер (подача деталей по одной в случайной 3D-ориентации)
  ├── Vision Sensor RGB 640×640 (над конвейером) — /vision_sensor_main
  ├── Vision Sensor Depth 640×640 (над конвейером) — /vision_sensor_depth
  │     (аналог Intel RealSense D435i, режим depth buffer)
  └── Зоны складирования (3 bins: gaika, vilka, vtulka)

Pipeline сортировки:
  RGB-кадр 640×640          Depth-кадр 640×640
    ↓                              ↓
  Детектор (CNN)           DepthEstimator
    Вход: 640×640×3          → height_m (высота верхней грани)
    Выход: bbox + class
    ↓
  Yaw-регрессор (CNN)
    Вход: crop детали (64×64 или 128×128)
    Выход: yaw угол (градусы)
    ↓
  GraspPlanner.resolve(class, height_m, yaw)
    → читает grasp_presets.yaml → GraspParams(pos, euler)
    ↓
  Pixel → World координаты
    ↓
  IK solver → движение манипулятора → захват с tilt → сортировка
```

## ПРИНЯТЫЕ РЕШЕНИЯ (НЕ ПЕРЕСМАТРИВАТЬ)

1. **Конвейер + подача по одной** — не bin picking. Bin picking = depth sensor + point cloud + 6DoF pose estimation в куче. За рамками scope. Упомянуть как перспективу в заключении.

2. **Детали спавнятся в случайной 3D-ориентации** — падают физикой CoppeliaSim и ложатся на произвольную грань. Это усложняет задачу: нет фиксированной высоты захвата.

3. **Depth-сенсор определяет высоту** — second Vision Sensor в режиме depth buffer. Аналог Intel RealSense D435i. `DepthEstimator` находит верхнюю грань по 5-му перцентилю глубины в bbox. Без дополнительной CNN.

4. **GraspPlanner + grasp_presets.yaml** — при смене деталей изменяется только YAML. Код планировщика универсален. Пресеты задают: height_min/max, tilt_deg, contact_offset для каждой стабильной позы.

5. **Захват с наклоном (tilt)** — для сложных поз (вилка на основании, детали на боку) gripper наклоняется до 40°. IRB 140 оси 4-5 обеспечивают нужный tilt. GRASP_Z_OFFSET/XY_OFFSET сохранены как fallback.

6. **Bbox + yaw** — не полный 6DoF pose. Высота из depth-сенсора + yaw из регрессора = достаточно для захвата.

7. **Кастомный CNN** — не готовая YOLO. Архитектура детектора и yaw-регрессора пишутся с нуля, слой за слоем, на PyTorch. Это усиливает вклад автора и научную новизну.

8. **Две раздельные сети** — детектор (bbox + class) и yaw-регрессор. Отлаживаются и обучаются независимо.

9. **Остановка конвейера** на время захвата — не tracking в движении.

10. **Sim-to-sim валидация** — sim-to-real обсуждается теоретически + domain randomization как мост.

11. **PyTorch** для обеих сетей.

12. **6-осевой IRB 140 обоснован:**
    - Yaw-ориентация захвата (ось 6)
    - Наклон tilt для захвата с боковых граней (оси 4, 5)
    - Разные зоны складирования с разной высотой
    - SCARA — не обеспечит tilt и ориентацию
    - IRB 140 — промышленный стандарт, воспроизводимость

## НАУЧНАЯ НОВИЗНА

1. Кастомная архитектура свёрточного детектора, оптимизированная для задачи детекции промышленных деталей на синтетических данных
2. Методика замкнутого цикла генерации синтетических данных в CoppeliaSim с автоматической аннотацией через vertex-transform проекцию 3D→2D при произвольной ориентации деталей и domain randomization
3. Количественный ablation study влияния факторов domain randomization на метрики детекции и accuracy сортировки
4. Двухступенчатый pipeline «детекция + yaw-регрессия + оценка высоты из depth» для планирования захвата деталей в произвольной позе
5. Конфигурируемый планировщик захвата (grasp_presets.yaml): при смене деталей изменяется только конфигурация, код остаётся нетронутым

## ОЖИДАЕМЫЕ ВОПРОСЫ НА ЗАЩИТЕ

1. Почему кастомная сеть, а не YOLO? → Полный контроль над архитектурой, оптимизация под задачу, глубокое понимание работы детектора, вклад автора
2. Почему 6-осевой, а не SCARA? → Ориентация захвата (yaw), наклон tilt для боковых поз, разные зоны складирования
3. Почему синтетика? → Автоаннотация, масштабируемость, воспроизводимость; детали в произвольной 3D-ориентации
4. Sim-to-real? → DR сужает gap, fine-tuning на реальных данных
5. Почему две сети, а не одна? → Независимая отладка, чистый ablation study, модульность
6. Зачем depth-сенсор? → Детали ложатся на разные грани, высота непостоянна. Depth дешевле второй камеры и не требует CNN
7. Зачем grasp_presets.yaml? → Разделение данных и кода: смена деталей = только YAML; масштабируемость
8. Bin picking? → Перспектива, требует point cloud + 6DoF pose estimation
